The general pattern that object creation function parameters in Vulkan follow is:

    1. Pointer to struct with creation info
    2. Pointer to custom allocator callbacks, always ignored in this tutorial as it is optional if you’re using the default.
    3. Pointer to the device, instance, or context on which the constructor depends
    4. Returns the pointer to the raii constructed object.


A way to check for errors in Vulkan is with .has_value() (which returns something similar to HRESULT), or try -> catch. (try runs the code inside the block, if there's an error with it, catch the error and throw a warning message -- exception.)

VUlkan doesn't have a "default framebuffer", so it requires an extension infrastructure that owns the buffers we're rendering before we visualize them on the screen.
This infrastructure is known as the "SWAP CHAIN" and as a result of it not being built in by default, we have to explicitly create it.

The swap chain is a queue of images that are waiting to be presented to the screen
    The process is our application creates an image, and then it returns it to the queue.

    The exact way the queue works, and the conditions for presenting the images from a queue depending on how the swap chain is set up.
    However, the general purpose of the swap chain is to synchronize the presentation of the images with the refresh rate of the screen


The first step to begin creating the swap chain is to check if our graphics card has support for the swap chain
    Some graphics cards are designed for servers, and thus don't have any display output, for example.

Due to image presentation being tied to the window system and its surface, swap chain isn't part of the vulcan core, and so we need to enable the device extension VK_KHR_swapchain (after checking if our device supports it)

Buffers are blocks of memories on the GPU that store data for the GPU to read/write.
https://docs.vulkan.org/tutorial/latest/03_Drawing_a_triangle/02_Graphics_pipeline_basics/00_Introduction.html for a neat visual
The graphics pipeline is the sequence of operations that takes everything from the vertices and textures of meshes, to pixels in the render target. It follows..
Vertex/Index buffer => Input Assembler -> Vertex Shader -> Tesselation Shader -> Geometry Shader -> Rasterization -> Fragment Shader -> Color Blending => Framebuffer

The INPUT ASSEMBLER collects the raw vertex data from the buffer (sometimes collects index buffers to repeat certain elements without having to duplicate the vertex data)

The VERTEX SHADER runs for every vertex (generally applies transformations) to turn vertex positions from model space to screen space -- it also passes individual vertex data down the pipeline.

the TESSELATION SHADER subdivides geometry, which increases the quality to make things look less flat.

the GEOMETRY SHADER is similar to the tesselation shader (but somehow more flexible): it runs on every primitive (primitive being vertices: a single point, a line, a triangle) and discards or put more primitives in to make it neater.

the RASTERIZATION stage turns the primitives into fragments. Fragments are the pixel elements that will eventually be displayed on the framebuffer -- Fragments outside of the screen (canvas) are discarded.
    Usually, if clipped=true, fragments behind other fragments are also discarded because of depth testing.

the FRAGMENT SHADER is called upon EVERY fragment that wasn't discarded and determines which framebuffers the fragments are written to and with which colour and depth values.
    It does this from the data from the vertex shader from earlier, where the data specifies texture coordinates and lighting.

the COLOR BLENDING stage applies operations to mix different fragments together that map to the same pixel within the framebuffer -- fragments can overwrite each other, add up, or be mixed due to transparency.



the Vertex Shader, Tesselation Shader, Geometry Shader, and Fragment Shader are programmable, which means we use our own code onto the graphics card to apply the exact operations we want.
    These programs run on many GPU cores simulatenously to process many objects in parallel, like vertices and fragments.
    For example, this allows us to use Fragment Shaders to implement anything from texturing and lighting, to ray tracers.

whereas the Input Assembler, Rasterization, and Color Blending are fixed function stages: we can just tweak their operations using parameters, but they way they work is predefined.
    With older APIs like OpenGL and direct3D, these were able to be programmable, but Vulkan makes it immutable, so you'd have to recreate the pipeline from scratch if you want to do custom stuff.

Some of the programmable stages are optional based on what we're trying to do.
    For example, tesselation and geometry shaders can be disabled if we're just drawing simple geometry.
    Another example: if we're shadow mapping, we can ditch the fragment shader.



The way colour blending works IN THE FIRST WAY (1. Mix the old and new colour to make a final colour) is shown fairly well with this pseudocode (if blendEnable is false, it's simply unmodified)

In previous versions of vulkan (pre 1.3), we'd need to create framebuffers to "bind" our image views to a render pass.
However, w/ the introduction of Dynamic Rendering in Vulkan 1.3, we're now able to render directly to image views without creating framebuffers or render passes.
As a result, dynamic rendering simplifies the process by eliminating the need for a render pass and framebuffer objects.
 Instead, we specify color, depth, and stencil attachments directly when we begin rendering
 We do this through structs vk::RenderingAttachmentInfo and vk::RenderingInfo to specify our attachments and rendering options.


Rendering a frame in Vulkan follows these steps:

1. Wait for the previous frame to finish
2. Acquire an image from the swap chain
3. Record a command buffer which draws the scene onto that image
4. Submit the recorded command buffer
5. Present the swap chain image


Operations w/ Vulkan are executed simulatenously, but they are asynchronous. So sometimes the functions will return before the operation has finished

We need to explicitly declare the order of operations as they happen on the GPU, some operations include:
Acquire an image from the swap chain; Execute commands that draw onto the acquired image; Present that image to the screen for presentation, returning it to the swap chain

These events (operations) are set in motion with a single function call, but they are executed asynchronously.
The function calls will return before the operations are actually finished (because the function is handing it off to the GPU for execution via a queue) and the order of execution is also undefined.
    Which is bad because operations typically depend on the previous one finishing (see the barrier we made -- pipelineBarrier2)
So we use fences and semaphores to dictate the ordering.


A Semaphore is used to add order between queue operations on the GPU.
    Queue operations refers to the work we submit to a queue, either from a command buffer or from functions (which is shown later) -- some examples of queues are the graphics queue and presentation queue.
    Semaphores are used to order work onto queues -- the submissions of works can be directed to the same queue, or different queues.
        In Vulkan, there are two kinds of semaphores: binary and timeline -- the tutorial only speaks about Binary semaphores, not timeline. (so from this point forward, mentioned semaphores refer exclusively to binary)
    A binary semaphore is either unsignaled or signaled.
        It begins as unsignaled.
        We use binary semaphores to order queue operations by providing the semaphore within a queue submission as a 'Signal' and as a 'Wait' in another queue operation.
            The way this works is whenever queue operation A finishes executing, we 'signal' the semaphore, and operation B will wait for the semaphore to be signaled before executing.
                For example, Semaphore S, Operation A and B (executed like A -> B ), whenever operation A is finished, it signals the Semaphore to be 'Signaled', then Operation B will wait and only run whenever semaphore has been Signaled.

Pseudocode demonstrating the example:
    vk::raii::CommandBuffer A, B = ... // record command buffers
    vk::raii::Semaphore S = ...        // create a semaphore

    // enqueue A, signal S when done - starts executing immediately
    queue.submit(work: A, signal: S, wait: None)

    // enqueue B, wait on S to be signal'd to actually start B -- this line WILL execute, but the command buffer's commands will wait for the signal (im just saying it'll put this command in a queue to be executed*)
    queue.submit(work: B, signal: None, wait: S)

*Just repeating what I said, but formally from the tutorial
    the calls to submit() return immediately, so the waiting ONLY occurs on the GPU (submit runs returns back to caller and the command's in the GPU's hands, waiting for signal -> continue as normally in the CPU code);
     the CPU continues without blocking or waiting for anything. To make the CPU wait, we use a different synchronization method: Fences.


A fence is similar to Semaphores, but on the CPU (as opposed to Semaphore working on the GPU) -- CPU is formally known as the host (because it's where you're writing code to, CPU code'll direct code onto the GPU).
    Fences are either in a signaled or unsignaled state, too. Whenever we submit work to be executed, we can attach a fence to that work, and whenever the work finishes, the fence will be signaled;
     in turn, we can make host/cpu code wait for the fence to be signaled, which guarantees that the work'll finish before the host continues.

Pseudocode demonstrating fences:
    vk::raii::CommandBuffer A = ... // record command buffer with the transfer
    vk::raii::Fence F = ...         // create the fence

    // enqueue A, start work immediately, signal F when done (the CPU continues right after as .submit() returns instantly, regardless of signal, stop occurs when we hit device.waitForFences(F))
    queue.submit(work: A, fence: F)

    device.waitForFences(F) // blocks execution until A has finished executing (the host stalls here till signal)

    save_screenshot_to_disk() // can't run until the transfer has finished (for the sake of the example, we need to make sure the screenshot was taken and transferred before we save)

However, generally it's not ideal to block the host unless necessary as we want to feed them more with useful work. Ideally, we just use semaphores (or other synchronization primitives/objects we've not seen yet) to synchronize our work.
We also must reset the fence back to the unsignaled state because fences control the execution of fences, and so the host gets to decide when to reset the fence -- semaphores, on the other hand, don't need to manually reset it as the host is not involved (goes to the GPU).

In summary, semaphores are used to specify the execution order of operations on the GPU while fences are used to keep the CPU and GPU in sync with each-other.

    void createVertexBuffer()
    {
        // You should know this by now, but I'm just regurgitating it.
        // A buffer is a chunk of memory, which starts off empty, but gets populated by some arbitrary data. We're making a buffer here to store vertex data, and the reason we're allocating buffer memory ourselves here is because it isn't automatic.
            // We are responsible for memory management, including buffers.

        vk::BufferCreateInfo bufferInfo {
            .size        = sizeof( vertices[0] ) * vertices.size(), // .size sets the buffer's reserved memory in bytes: gets the size of a single vertex in bytes, and multiply it by the amount of vertices in the container.
            .usage       = vk::BufferUsageFlagBits::eVertexBuffer,  // .usage specifies which purpose the data inside of the buffer will be used for -- you can specify multiple purposes with bitwise OR.
            .sharingMode = vk::SharingMode::eExclusive              // similarily with images inside of the swap chain, buffers can be owned by an exclusive queue family, or concurrently be owned by multiple queue families. We're only using the eGraphics queue, so it's exclusive.
        };      // Note: we are NOT assigning the queue family here, we're just saying it'll be exclusive/owned by a single queue.
            // There's also a .flags member, which is used to configure sparse buffer memory. We're not needing it right now, so we're leaving it as be, defaulted to 0.

        // and similiarily with a bunch of other constructors for Vulkan objects, give the device and the information we just specified above (buffer info) to actually create a vertex buffer.
        vertexBuffer = vk::raii::Buffer( logicalDevice, bufferInfo );
            // an IMPORTANT notice: for GPU buffers, you NEED to explicitly allocate memory towards it, unlike with CPU's commandPool + commandBuffers -- it is automatic after object creation.
            // right now, all we have is a handle to a vertexBuffer which information of what is supposed to be inside of it: we need to actually allocate the memory inside the GPU.

        // The first step to ACTUALLY allocate memory inside the GPU is to get the memory requirements of the buffer. (REMEMBER: THIS IS ONLY NEEDED FOR GPU BUFFERS)
        // struct vk::MemoryRequirements contains three member variables:
            // .size: the size of the required memory in bytes (it MAY differ from bufferInfo.size)
            // .alignment: the offset in bytes from where the buffer actually begins in the allocated region of memory (depends on bufferInfo's .usage and .flags)
                //        this does NOT mean it shrinks the buffer memory size to "pad", it just shifts the address by X bytes to then begin the buffer itself at that shifted address
            // .memoryTypeBits: the bit field of memory types that are suitable for the buffer -- this is NOT specified by .usage and isn't manually set, vulkan sets this automatically
                // graphics card themselves offer different types of memory to allocate from, and each type of memory varies in terms of allowed operations and performance characteristics, and it's up to us to find the right memory type.
        vk::MemoryRequirements memoryRequirements = vertexBuffer.getMemoryRequirements();
        // Then properly fill out the Info struct with whatever requirements.
        vk::MemoryAllocateInfo memoryAllocateInfo {
            .allocationSize  = memoryRequirements.size, // the byte size to actually allocate for our buffer
            .memoryTypeIndex = findGPUBufferMemoryType( memoryRequirements.memoryTypeBits, vk::MemoryPropertyFlagBits::eHostVisible | vk::MemoryPropertyFlagBits::eHostCoherent )
                // for the first parameter of findGPUBufferMemoryType(), we're just passing the basic buffer memory requirements (NOT including the special properties we want, which is param2) to find the barebones support for this buffer.
                // the second param is the special properties we require of the memory type -- we want to map the memory type to the CPU for access to our vertices, hence vk::MemoryPropertyFlagBits::eHostVisible
        };

        // Then the second step is to construct vk::raii::DeviceMemory to actually allocate the memory within the GPU!
        // at this point, we've allocated memory onto the GPU, but the vertexBuffer object isn’t using it yet, hence vk::raii::Buffer::bindBufferMemory
        vertexBufferMemory = vk::raii::DeviceMemory( logicalDevice, memoryAllocateInfo );
        // first parameter specifies the memory handle to 'bind' the buffer to -- where this buffer's data exists.
        // second parameter is the offset within this region of memory (just keep it zero) -- if the offset is non-zero, then it is required to be divisible by memRequirements.alignment
        vertexBuffer.bindMemory( *vertexBufferMemory, 0 ); // Without this, vertexBuffer is NOT referencing ANY memory, and thus won't be useful.

        // a pointer to our vertex buffer's memory (left of .mapMemory) -- we need to do this to actually grab ahold of the vertex data, otherwise it's inaccessible.
        // first parameter is the offset (here, zero -- we want the whole thing), and the second parameter is the byte size of the region (within vertexBufferMemory) we want to grab ahold of (here, size of the entire butter -- we want the whole thing)
        // think of it as a rectangular window: we start at the first point (offset 0, first param) of the rectangle, and cover the entire area of the rectangle (.size of the buffer, second param)

The current vertex buffer we have (the one thats created from the function above) works,
but the memory type that us to access it from the CPU may not be the most optimal memory type for the graphics card to read data itself.
The most optimal memory has the VK_MEMORY_PROPERTY_DEVICE_LOCAL_BIT flag, and the memory with this flag enabled USUALLY doesn't allow the data to be accessed by the CPU.

So, we create TWO vertex buffers:
    1. something called a "Staging Buffer" that utilizes a memory type that allows the CPU to access its memory/data from within the GPU;
    2. the buffer that houses the data with a memory type that uses the most optimal property flag to allow the GPU to read from this optimized buffer WITHOUT using a slower memory type that'll allow CPU communication.
The way this works is we assign the staging buffer whatever data we want the GPU to store from the CPU,
and then use a buffer copy command to move the staging buffer's data into the second, optimized buffer -- staging buffer is then destroyed after sending its contents to the optimized buffer.
    Essentially, a staging buffer is a mule.

The buffer copy command requires a specific queue family: VK_QUEUE_TRANSFER_BIT
However, any queue family with the eGraphics or eCompute flag already implicitly supports VK_QUEUE_TRANSFER_BIT, so we don't need to explicitly list it within our vk::QueueFlags requiredQueues object.


See https://docs.vulkan.org/tutorial/latest/04_Vertex_buffers/02_Staging_buffer.html#_conclusion for some handy info about allocation buffers -- might come in handy later.

It's also recommended to store multiple GPU buffers (including vertex, index) from a single allocated memory pool.
Furthermore, it's recommended by driver devs to make a single general GPU buffer to house/store other GPU buffers (inc. vertex/index),
and use offsets in commands like vkCmdBindVertexBuffers to differentiate the individual buffers within the general buffer by regions
    We're not literally putting a buffer inside a buffer, but designating each region of the general buffer's allocated space to house another buffer, and using offset commands to specify what individual 'buffer' we want.



A 'Resource Descriptor' is a way for shaders in the GPU pipeline to access whatever other resources in the GPU that it cant normally see, like GPU buffers or images.
A 'Uniform Buffer' is a buffer in GPU memory that stores whatever data we want to give the shaders in the GPU pipeline -- it is not exclusive to contain only vertices, indices, etc
    The resource descriptor and uniform buffer goes hand-in-hand: the uniform buffer houses the actual data, and the resource descriptor allows the GPU pipeline shader functions to access the data in the uniform buffer.


So, first step is to make a uniform buffer that contains transformation matrices (we're making 3D stuff now) and have the vertex shader access them through the resource descriptor.

The usage of a resource descriptor consists of three parts:
    1. Specify a 'descriptor set layout' during pipeline creation
    2. Allocate a 'descriptor set' from a descriptor pool
    3. Bind the 'descriptor' set during rendering

A 'Descriptor Set Layout' specifies the types of resources that'll be accessed by the shaders (in the pipeline) (similarily w/ how we specify what color formats will be used for the rendering)
    A Descriptor Set Layout contains multiple slots, and these slots are housed by individual Descriptor Set Layout Bindings

A 'Descriptor Set Layout Binding' are the specific type of resources that are referenced by the Descriptor Set Layout

A 'Descriptor Set' specifies the actual uniform buffer or image resources that'll be bound to the descriptors (where the data is housed) (similarily w/ how the image views gets binded to the render's attachments)
